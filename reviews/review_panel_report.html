Pre-Lock Caveats

BLOCKER 1. Enforce append-only at the database (closes P1-A)
▸

On both floor_conv_headers and floor_conv_messages:

REVOKE TRUNCATE, REFERENCES, TRIGGER FROM anon, authenticated, service_role;

-- Statement-level TRUNCATE guard (both tables)
CREATE OR REPLACE FUNCTION immutable_floor_guard() RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  RAISE EXCEPTION 'immutable floor: % not permitted', TG_OP;
END;
$$;
CREATE TRIGGER floor_headers_no_truncate BEFORE TRUNCATE ON floor_conv_headers
  FOR EACH STATEMENT EXECUTE FUNCTION immutable_floor_guard();
CREATE TRIGGER floor_messages_no_truncate BEFORE TRUNCATE ON floor_conv_messages
  FOR EACH STATEMENT EXECUTE FUNCTION immutable_floor_guard();

-- Row-level UPDATE/DELETE guard (both tables)
CREATE TRIGGER floor_headers_immutable BEFORE UPDATE OR DELETE ON floor_conv_headers
  FOR EACH ROW EXECUTE FUNCTION immutable_floor_guard();
CREATE TRIGGER floor_messages_immutable BEFORE UPDATE OR DELETE ON floor_conv_messages
  FOR EACH ROW EXECUTE FUNCTION immutable_floor_guard();

Model on Supabase's own protect_objects_delete / protect_buckets_delete triggers (already in storage schema — this pattern is supported).

Proof test: TRUNCATE floor_conv_messages as service_role → must raise exception. Capture output in lock artifact.

Severity Register

P1 Append-only enforcement absent LIVE-VERIFIED
▸

anon, authenticated, service_role, and postgres all hold TRUNCATE on both floor tables. Additionally: anon/authenticated/service_role hold REFERENCES + TRIGGER. Zero guard triggers. RLS enabled but has zero policies and doesn't gate TRUNCATE.

Evidence: live role_table_grants, pg_trigger (5 non-internal triggers, none on floor tables), pg_class.relrowsecurity.

RESOLVED FK cascade (feared third bypass) LIVE-VERIFIED
▸

Phase 8 raised concern that fk_header might be ON DELETE CASCADE (silent message deletion on header delete). Query: SELECT confdeltype FROM pg_constraint WHERE conname='fk_header' → 'a' = NO ACTION.

The FK protects the floor — a header with child messages cannot be deleted. Feared bypass does not exist.

P2 "Verbatim" claim inaccurate — JSONB key reorder + scrub-v1 LIVE-VERIFIED
▸

JSONB: 200/200 content_blocks sampled: all key-reordered, 0 byte-identical, 0 value loss. JSONB canonically sorts keys alphabetically and silently collapses duplicate keys (last-wins).

scrub-v1: 273 credential redactions; 8 touch thinking-block paths (RTSP, anthropic, postgres secrets inside thinking text). verify.log: passed=true, 0 residual hits over 331 MB.

TEXT columns (including text, created_at) are byte-verbatim.

Remediation: relabel (Caveat 4). No data loss; labeling fix only.

P2 pgvector absent vs "+pgvector" in title LIVE-VERIFIED
▸

pg_extension confirms no vector/pgvector. pg_available_extensions: vector 0.8.0 is installable but not installed. Installed extensions: pg_stat_statements, pgcrypto, plpgsql, supabase_vault, uuid-ossp.

Remediation: drop "+pgvector" from title or annotate as deferred sidecar (Caveat 4). Not required for storage-only lock.

P2 Multi-snapshot mixed-version query (pre-snapshot-2) LIVE-VERIFIED structurally
▸

PK is (snapshot_id, conv_uuid, msg_uuid). After snapshot 2 ingest, a naive query WHERE conv_uuid = X returns messages from both snapshots with no indicator. The current_snapshots view / selection policy must be defined before snapshot 2 is ingested.

Not blocking for current single-snapshot lock. Required before snapshot 2 (Caveat 6).

P3 No secondary indexes — all tree traversal is Seq Scan LIVE-VERIFIED
▸

EXPLAIN: parent_message_uuid filter → Seq Scan on floor_conv_messages (22,801 rows). Only PK btrees exist. Post-lock recommendation: add (snapshot_id, conv_uuid, parent_message_uuid) index for tree traversal.

Post-Lock Recommendations (not blocking)

1. Off-site immutable ndjson copy

Object storage with object-lock (e.g., S3). Highest-priority post-lock: the ndjson-canonical reframe makes the ndjson the SPOF. Single-copy canonical artifact is a latent failure.

2. Verify/enable PITR

Or formally accept ndjson-rebuild as the backup strategy and document the expected RTO.

3. Pre-ingest lint

Lint future ndjson exports for duplicate keys and null bytes before ingest. Postgres TEXT rejects U+0000; JSONB silently collapses duplicate keys.

4. Secondary index

(snapshot_id, conv_uuid, parent_message_uuid) for tree traversal. Current plan requires full seq scan of 22,801 rows for any tree query.

5. CHECK constraints

CHECK (sender IN ('human','assistant')), is_root↔sentinel agreement, timestamp format — added carefully to never rewrite stored bytes.

6. Single-transaction ingest for snapshot #2

All-or-nothing ingest (single transaction or staging-swap pattern) to handle partial failures without requiring an append-only-violating DELETE.

7. Header-flag integrity drill

Re-derive multi_root, has_branches, message_count from current tree and confirm they match denormalized header values. Clean today; no enforcement for future snapshots.
